# 🏆 دليل مشروع التخرج: نظام إدارة أكاديمية VFC لكرة القدم
## (VFC Football Academy Management System)

---

## 📌 1. نبذة عامة عن المشروع (Project Overview)
**نظام VFC Academy** هو منصة سحابية إدارية ومالية متكاملة (SaaS Dashboard) مُصممة خصيصاً لإدارة الأكاديميات الرياضية متعددة الفروع.
يتميز النظام بمعالجة العمليات المالية المعقدة، مثل: **تحديد أيام تقفيل مالية مختلفة لكل فرع**، حساب المديونيات التراكمية بشكل لحظي، إدارة الرواتب والسلف للمدربين، تتبع مخزون ومبيعات الأطقم الرياضية، وتتبع حضور اللاعبين مع الربط الفوري بخدمة WhatsApp لإرسال التنبيهات لأولياء الأمور.

---

## 🛠️ 2. التقنيات المستخدمة (Technical Architecture & Tech Stack)

### **أولاً: الواجهة الأمامية (Frontend Framework)**
* **React 19 (TypeScript)**: بناء واجهة مستخدم تفاعلية (SPA) قائمة على المكونات (Component-Driven Architecture) مع ضمان دقة الأنواع (Type Safety).
* **Vite**: بناء سريع وحزم خفيفة للواجهة (Build Tool & Dev Server).
* **TailwindCSS (v4)**: نظام تصميم عصري، مرن، وداعم كلياً للغة العربية والتصميم من اليمين لليسار (RTL/LTR Dynamic Layouts).
* **Lucide React Icons**: مكتبة الأيقونات التفاعلية.
* **Recharts**: رسم المخططات والرسوم البيانية التفاعلية للإيرادات وتطور الأداء المالي.
* **jsPDF & html2canvas**: توليد وإصدار إيصالات الدفع وتقارير PDF فورية قابلة للطباعة والتنزيل.
* **SheetJS (XLSX)**: تصدير بيانات اللاعبين والتقارير المالية والمديونيات إلى ملفات Excel.

---

### **ثانياً: القاعدة الخلفية وقاعدة البيانات (Backend & Database)**
* **Supabase (PostgreSQL 17 Database)**:
  * **Database Engine**: قاعدة بيانات PostgreSQL متطورة تدعم العلاقات والأدوار المعقدة.
  * **Supabase Realtime**: محرك التحديث اللحظي باستخدام بروتوكول Websockets لإعادة تحميل البيانات تلقائياً على كل الشاشات المفتوحة دون الحاجة لعمل Refresh.
  * **Row Level Security (RLS)**: حماية البيانات على مستوى الصفوف في قاعدة البيانات بناءً على هوية ودور المستخدم (`auth.uid()`).
  * **Stored Procedures & Remote Procedure Calls (RPC)**: كتابة الدوال المعقدة بلغة SQL داخل قاعدة البيانات مباشرة لتحقيق أعلى كفاءة وأسرع أداء (High Performance Data Processing).
  * **Authentication**: نظام مصادقة آمن يعتمد على رموز JWT (JSON Web Tokens).

## 📐 3. هيكل قاعدة البيانات والجداول (Database Schema)

1. **`branches` (الفروع):**
   * تخزين بيانات الفرع، اسم الفرع، اللون الخاص (`accent_color`)، و**يوم التقفيل المالي للفرع (`closing_day`)**.
2. **`users` (المستخدمين والأدوار):**
   * يمتد من جدول `auth.users` ويخزن الأدوار: `owner` (صاحب الأكاديمية)، `admin` (مدير فرع)، `coach` (مدرب).
3. **`groups` (المجموعات والفرق):**
   * ربط اللاعبين بالمدرب المسؤول وأيام ومواعيد التدريبات الأسبوعية.
4. **`players` (بيانات اللاعبين):**
   * بيانات اللاعب، الكود الفريد (`player_code`)، تاريخ التسجيل، نوع الاشتراك (شهري/دوري)، قيمة الاشتراك الشهري (`fee_amount`) وقيمة اشتراك الدوري (`fee_amount_periodic`).
5. **`attendance` (الحضور والغياب):**
   * تتبع الحضور اليومي (`present`, `absent`, `late`) مع التأكد من عدم تكرار تسجيل اللاعب في نفس اليوم (`UNIQUE(player_id, session_date)`).
6. **`payments` (المدفوعات والاشتراكات):**
   * سجل المقبوضات المالية شامل المبلغ، طريقة الدفع (نقدي/تحويل)، الفترة التغطوية، وتاريخ السداد.
7. **`expenses` (المصروفات التشغيلية):**
   * المصروفات مقسمة حسب الفئات (`salaries`, `equipment`, `rent`, `utilities`, `kits_stock`, `other`).
8. **`coaches` & `coach_salary_payments` & `coach_advances` (إدارة المدربين):**
   * الراتب الأساسي، السلف المستقطعة، ودفعات الرواتب الشهرية.
9. **`kit_items` & `kit_purchases` (مبيعات المتجر والأطقم):**
   * إدارة مخزون الأطقم، سعر التكلفة، سعر البيع، تنبيهات حد النقص (`low_stock_threshold`)، ومبيعات اللاعبين.

## ⚙️ 4. الخوارزميات والمعالجة البرمجية المميزة (Advanced Core Logic)

### **1. خوارزمية الشهر المالي حسب يوم تقفيل الفرع (`fn_date_to_financial_month`)**
* **المشكلة:** بعض الفروع تقفل حساباتها يوم 20 في الشهر، وبعضها يوم 30 أو 31.
* **الحل البرمجي (SQL Function):**
```sql
CREATE OR REPLACE FUNCTION fn_date_to_financial_month(p_date date, p_closing_day integer)
RETURNS text LANGUAGE sql IMMUTABLE AS $$
  SELECT to_char(
    CASE 
      WHEN p_closing_day IS NULL OR p_closing_day <= 0 OR p_closing_day >= 31 THEN p_date
      WHEN EXTRACT(DAY FROM p_date) > p_closing_day THEN p_date + INTERVAL '1 month'
      ELSE p_date
    END,
    'YYYY-MM'
  );
$$;
```
* **الشرح:** إذا كان يوم التاريخ المدخل أكبر من يوم تقفيل الفرع، يتم ترحيل الدفعة/المصروف تلقائياً كإيراد للشهر المالي القادم.

---

### **2. دالة حساب كشف المديونيات الذكي (`rpc_debt_list`)**
* تقوم بحساب عدد الأشهر المنقضية منذ تاريخ تسجيل اللاعب بناءً على يوم تقفيل فرعه.
* تحسب الإجمالي المتوقع (شهري ودوري)، وتقارنه بإجمالي المدفوعات المسجلة للاعب، وتحدد مديونية الشهري ومديونية الدوري وتاريخ التجديد القادم فوراً.

---

### **3. دفتر اليومية المالي الموحد (`rpc_get_ledger`)**
* دالة تقوم بعمل `UNION ALL` بين 5 مصادر مالية متفرقة:
  1. اشتراكات اللاعبين (إيراد)
  2. مبيعات الأطقم (إيراد)
  3. المصروفات التشغيلية (مصروف)
  4. سلف المدربين (مصروف)
  5. دفعات رواتب المدربين (مصروف)
* تُخرج الحركات مرتبة زمنيًا لتقديم تقرير مالي موحد ودقيق.

## 🎓 5. الأسئلة المتوقعة في مناقشة مشروع التخرج وإجاباتها (Defense Q&A)

### **س1: لماذا اخترتم React + TypeScript بدلاً من JavaScript العادية؟**
> **الإجابة:** TypeScript تقدم نظام Types قوي يقلل أخطاء البرمجة (Runtime Errors) أثناء تطوير الأنظمة المالية، وتوفر كوداً أكثر استقراراً وسهولة في الصيانة (Maintainability) واكتشاف أخطاء التحويلات قبل تشغيل الكود.

---

### **س2: كيف تضمنون أمان البيانات وعدم وصول المدرب أو المدير لبيانات مالية لا تخصه؟**
> **الإجابة:** نستخدم نظام **Row Level Security (RLS)** المدمج في قاعدة بيانات PostgreSQL عبر Supabase. البيانات لا تُحمي فقط من الواجهة الأمامية، بل يتم التحقق من شريحة الأمان والدور (`user_role`) مباشرة داخل قاعدة البيانات قبل تنفيذ أي استعلام (Query Execution).

---

### **س3: كيف تتعامل المنظومة مع تحديث البيانات اللحظي بين الأجهزة المختلفة؟**
> **الإجابة:** نستخدم **Supabase Realtime API** القائم على WebSockets. عند إضافة أي خطوة (مثل تسجيل دفعة أو غياب)، يتم إرسال تنبيه لحظي لجميع الأجهزة المشتركة في القناة (`Postgres Changes Replication`) لتحديث الشاشة فوراً دون الحاجة لإعادة تحميل الصفحة.

---

### **س4: كيف يتم التعامل مع اختلاف أيام التقفيل المالي للفروع؟**
> **الإجابة:** قمنا بكتابة دالة مخصصة داخل قاعدة البيانات (`fn_date_to_financial_month`) تأخذ تاريخ الحركة ويوم تقفيل الفرع، وإذا تجاوز التاريخ يوم التقفيل، تُحسب الحركة تلقائياً على الشهر المالي التالي لضمان دقة التقارير المالية لكل فرع بشكل مستقل.

---

### **س5: ما الذي يميز هذا النظام عن الأنظمة المتاحة في السوق؟**
> **الإجابة:** 
> 1. المرونة المالية العالية بتخصيص أيام التقفيل لكل فرع.
> 2. الربط المباشر مع تطبيق WhatsApp لتنبيه أولياء الأمور تلقائياً.
> 3. السرعة الفائقة بفضل المعالجة المباشرة داخل قاعدة البيانات (Database Stored Procedures).
> 4. الواجهة التفاعلية الفاخرة المجهزة بالكامل باللغة العربية والتصميم الموجه للشرق الأوسط.
